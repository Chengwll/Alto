Section 01

概念起源与背景

2026年初，AI Agent 的能力已远超预期——OpenAI Codex 团队仅3名工程师，配合 AI 智能体，在5个月内完成了100万行代码，全程无人为编写。然而，这一突破也暴露出一个核心问题：当 AI 能自动生成大量代码时，工程领域的真正瓶颈，已不再是"让 AI 写得更好"，而是"让 AI 跑得更稳、更可控"。

💡
概念提出 2026年2月5日，HashiCorp 联合创始人 Mitchell Hashimoto 首次明确提出 "Harness Engineering"（驾驭工程）术语。此后，OpenAI 在其百万行代码实验报告中正式采纳该术语，Martin Fowler 与 Birgitta Böckeler 联名撰文深度解读，Anthropic 随即发布《长时运行 Agent 的有效线束》，使该范式迅速成为2026年 AI 工程领域最受关注的方法论。
2023–2024
提示词工程（Prompt Engineering）时代
工程师聚焦于优化单次对话指令，通过精心设计 Prompt 提升模型输出质量。
2025
上下文工程（Context Engineering）兴起
Andrej Karpathy 提出 Context Engineering 比 Prompt Engineering 更重要，关注如何为 AI 组织和注入精准的上下文信息。
2026.02
Harness Engineering 正式提出
Mitchell Hashimoto 提出驾驭工程，随即被 OpenAI、Anthropic、Stripe 等顶尖团队验证和采纳，清华大学等研究机构发布专项研究报告。
2026.03–04
爆发期：全行业共识形成
业界形成六大行业共识，LangChain、Stripe Minions 等标志性实践案例涌现，Harness Engineering 成为 AI 工程师必备方法论。

Section 02

核心定义与本质

🎯
权威定义（清华大学 2026 研究报告） Harness Engineering 是围绕 AI 智能体设计和构建约束机制、反馈回路、工作流控制和持续改进循环的系统工程实践。它不优化模型本身，而是优化模型运行的环境。核心哲学八字："人类掌舵，智能体执行"。

核心等式

Agent = Model + Harness
Model
提供智能
理解、推理、生成
Harness
提供工程保障
约束、反馈、可靠性

形象类比

🎙️
提示词工程
对着马喊话：通过优化指令让 AI 产生更好的单次输出，但无法保证持续可靠性。
🗺️
上下文工程
给马看地图：为 AI 提供精准的信息输入，优化知识和上下文的组织方式。
🛣️
驾驭工程
修建高速公路和护栏：为 AI 构建一整套可控、可追溯的运行环境，确保安全到达终点。
🐴
核心哲学
套上马具（Harness）的目的不是限制能力，而是让骏马安全、高效地奔向目标。
⚡
关键洞见（Birgitta Böckeler） "为了获得更高的 AI 自主性，运行时必须受到更严格的约束。"—— 更强的自主性与更严格的约束并不矛盾，反而相互依存。

Section 04

六层架构体系（JavaGuide 权威版）

Harness 可拆解为六层递进架构，构成从"定义边界"到"兜底恢复"的完整闭环设计系统。

L1
信息边界层（Information Boundary）
定义 Agent 该知道什么、不该知道什么。通过角色定义与目标设定、无关信息裁剪、任务状态结构化组织，确保 Agent 在任何时候都运行在干净、相关的上下文中。
🎯 优先级最高
L2
工具系统层（Tool System）
定义 Agent 如何与外部世界交互。包含工具的选拔原则、调用时机控制、结果的提炼与反馈机制，以及工具白名单的动态注册管理。
🔧 能力扩展
L3
执行编排层（Execution Orchestration）
管理多步骤任务如何串联。让模型走完"理解目标→判断信息→分析→生成→检查"的完整执行轨道，支持 Plan-Execute 策略和确定性节点编排。
⚙️ 流程控制
L4
记忆与状态层（Memory & State）
解决长任务中间结果如何管理。独立管理当前任务状态、中间产物和长期记忆，防止系统混乱，支持会话间状态恢复与持久化。
💾 状态持久
L5
评估与观测层（Evaluation & Observability）
建立 Agent 如何知道自己做对了的机制。构建独立于生成过程的验证机制（如浏览器自动化、独立评估 Agent），实时追踪上下文利用率等关键指标。
📊 可观测性
L6
约束、校验与恢复层（Constraints & Recovery）
出错了怎么办。预设规则拦截错误，提供重试或回滚机制，通过 Linter + CI 门禁机械化执行架构约束，设置最大迭代次数与停滞容忍阈值。
🛡️ 优先级最高
💡
投入建议 不要试图一开始就搭建所有六层。应优先投入 L1（信息边界） 和 L6（约束与恢复），这两层的投入产出比最高，能最快带来可靠性提升。
🔩
Section 05

六大核心组件（LangChain 技术框架）

LangChain 将 Harness 视为智能体能力的放大器，解决裸模型四大硬伤：无记忆、不能执行代码、知识过时、无工作环境。Harness 的六大核心组件如下：

# 组件 核心作用 关键功能
01 文件系统 持久化存储与版本管理 存储中间结果、支持多 Agent 协作、结合 Git 实现版本追踪与回滚
02 Bash + 沙箱 代码执行与验证 运行代码、安装依赖、执行测试；沙箱保障安全隔离；实现"写→跑→修"自验证循环
03 AGENTS.md 长期记忆与知识注入 以文件形式存储领域知识、无需微调模型即可动态注入上下文、支持项目级规范与陷阱记录
04 Web Search + MCP 实时信息获取 突破模型知识时间限制；MCP 协议连接内部系统（数据库、监控工具）
05 上下文工程 上下文质量与效率管理 压缩历史信息避免"上下文腐烂"；动态加载/卸载任务相关技能；分层管理核心/工作/历史上下文
06 编排 + Hooks 多 Agent 协作与质量管控 任务分解、调度、结果聚合；Hooks 实现自动化检查（代码规范、安全审核）；支持动态层级化编排
🧠
System Prompt 是 Harness 的神经中枢 System Prompt 贯穿所有六个组件，定义 Agent 的角色边界与安全规则。模型决定"如何思考"，Harness 决定"看到什么、用什么工具、出错怎么办"。

AI 落地全链路六大核心（行业实战视角）

从 AI 产品落地的全链路视角来看，Harness Engineering 可拆解为覆盖"从用户输入到结果返回"全链路管控的六大核心能力。以下内容整合自知识库多篇实战文章。

核心能力 定义 关键难点与方案
Prompt Engineering 通过设计有效提示词引导模型输出预期结果 不同模型响应差异大需针对性适配；Prompt优化依赖经验迭代；模型版本迭代可能导致Prompt"退化"需持续维护
Structured Output 约束模型输出为程序可解析的结构化格式（如 JSON） JSON Schema 校验、Grammar-based Decoding（outlines 库）、框架内置 Parser（如 LangChain Output Parser）

Agent Loop & Tool Use 让 AI 通过调用外部工具执行实际操作，实现多步任务闭环 核心机制 ReAct（Reasoning + Acting）；常见问题：循环依赖需限深度、工具失败需重试+降级策略

Evaluation & Monitoring 通过持续评测和线上监控确保 AI 表现稳定可控 评测方法：人工/规则/AI评测(LLM-as-Judge)/自动化测试；监控指标：成功率、P50/P99延迟、Token消耗、错误分布

Section 06

四大护栏机制（OpenAI 实践框架）

基于 OpenAI Codex 团队实践，Harness Engineering 可归纳为四大核心护栏，这也是实践最广泛的工程化框架。

📚
① 上下文工程（AGENTS.md）
提供动态、简洁的智能体指南，避免静态文档过时。
  • 使用渐进式披露结构（约100行索引文件）
  • 每行规则对应一个历史失败案例
  • 命令优先，代码示例优于文字描述
  • 指向更详细的分层文档，按需加载
⛓️
② 架构约束（机械化执行）
通过分层依赖模型和 Linter 规则，硬性约束代码结构。
  • 分层依赖：Types→Config→Repo→Service→UI
  • 自定义 Linter，CI 阻断违规合并
  • Linter 错误消息包含修复指令
  • 选择"无聊"技术，约束解决方案空间
🔄
③ 反馈循环（闭环自验证）
实现"智能体审智能体"的自我检查机制。
  • Agent-to-Agent 代码审核机制
  • 自动化测试套件：失败时回传错误信息
  • 四步循环：规划→构建→验证→修复
  • 退出前强制执行检查清单
🧹
④ 熵管理（技术债务控制）
通过持续治理对抗代码库熵增。
  • 编码"黄金原则"，Linter 强制执行
  • 定期运行后台清理 Agent（垃圾回收）
  • "文档园丁代理"自动修复过时内容
  • 持续小额偿还技术债务，而非集中处理

护栏协同流程与工具链

四大护栏形成完整的协同控制链路：用户请求 → 上下文工程 → 架构约束 → Agent 执行 → 反馈循环 → 熵管理 → 返回结果。

护栏 核心组件 关键工具/脚本
上下文工程 AGENTS.md 活文档、按需检索 agent_context_retriever.sh、context_injector.py
架构约束 分层依赖模型、Linter 规则 architecture_linter.py、lint-deps.py、lint-quality.py
反馈循环 Agent-to-Agent Review、自动化测试 review_agent.py、feedback_loop.sh、generate_test_suite.sh
熵管理 技术债务检测、文档园丁 tech_debt_detector.sh、doc_garden_agent.py、Critic/Refiner 脚本
💡
最佳实践要点（知识库实战总结） 上下文窗口使用率控制在 60% 以下，避免噪声；动态检索根据任务类型（read/write/debug/refactor）选择性加载；连续失败超过阈值（如 5 次）暂停执行防止雪崩（熔断机制）；定期清理每 3600 秒执行垃圾回收。

Section 07

AI Agent 典型失败模式

了解 AI Agent 的典型失败模式，是设计有效 Harness 的前提。以下是业界总结的最常见失败场景：

失败模式 表现描述 Harness 解决方案
一步到位（One-shotting） Agent 在单次会话中试图完成过多功能，上下文耗尽后留下半成品代码 上下文窗口监控 + 任务分解编排
过早宣布胜利（Premature Victory） 完成约 20% 即标记完成，缺少端到端验证 退出前强制检查清单 + 端到端测试验证
过早标记功能完成 代码提交但未测试，功能不可用 代码测试比监控（check_code_test_ratio.sh）+ 强制测试门禁
复制坏模式 Agent 放大代码库中的技术债务，复制并传播坏模式 架构约束（Linter）+ 代码风格门禁
上下文焦虑 当上下文窗口超过 40% 时，Agent 表现退化，输出质量显著下降 40% 阈值告警 + 上下文压缩重置策略
无意义重试 Agent 陷入重复尝试同一方案的死循环，无法自主识别停滞 最大迭代限制 + 状态指纹比对
静默失败 Agent 输出看似正常实则错误的结果，且不发出任何警告信号 独立评估 Agent + 生成者-评估者分离
💡
40% 上下文阈值法则 当上下文窗口利用率超过约 40% 时，Agent 输出质量会显著下降，进入"Dumb Zone"（幻觉增多、兜圈子、代码质量低）。生产环境中应设置 40% 阈值告警，触发上下文压缩或任务交接，而非继续填入信息。

上下文重置策略（Anthropic 方案）

当 Agent 因上下文接近饱和而表现不佳时，Anthropic 采用"上下文重置"而非单纯压缩：将当前任务状态、已完成工作等结构化提取，然后启动一个全新的"干净"Agent，通过交接文档恢复状态。这类似于重启进程来解决内存泄漏问题。

Section 08

顶尖团队一线实践案例

O
OpenAI Codex 团队
百万行代码实验 · 架构约束实践
Codex 团队以3人小组完成了最具标志性的 Harness Engineering 实践。其核心方法论包括：地图式文档（精简100行索引 AGENTS.md 指向深层文档，渐进式披露）；机械化约束（自定义 Linter 配合 CI，强制架构分层不可违反）；可观测性集成（每个 git worktree 有独立可观测栈，实时追踪 Agent 状态）；主动熵管理（定期运行"垃圾回收" Agent，扫描偏差并自动提交重构 PR）；仓库即唯一事实源（所有状态存入 Git，无外部依赖）。
3人 工程师团队
100万行 代码生成量
5个月 完成时间
零行 人为编写代码
A
Anthropic
双重智能体架构 · 长时程任务管理
Anthropic 针对长周期任务中"跨会话失忆"的核心痛点，创造了双重智能体架构： 初始化 Agent（项目开始时运行，生成详细需求列表 JSON、初始化脚本、进度文件，并进行初始 Git 提交）+ 编码 Agent（后续会话中运行，每次先读进度文件和 Git 历史，约束为"每次只处理一个明确功能"，完成后立刻提交并更新进度）。此外，Anthropic 倡导随着模型能力提升，应定期简化 Harness，避免过度工程化。
✓ 跨会话状态持久化
✓ Planner+Generator+Evaluator
4h 连续可靠编码时长
S
Stripe Minions
高度自动化无人值守 · 混合状态机编排
Stripe 的 Minions 项目代表了 Harness Engineering 在生产环境的极致实践。核心模式为混合状态机编排：该确定的地方（如 lint、格式化、类型检查）由确定性代码执行；该灵活的地方（如实现具体功能、修复 Bug）交给 Agent。这种"确定性节点夹住概率性工作"的模式，实现了高度自动化的无人值守运行。
1300+ 每周自动合并 PR 数
无人值守 运行模式
L
LangChain
纯 Harness 优化 · 性能质的飞跃
LangChain 团队通过一次震撼业界的实验，证明了 Harness 优化的价值：未改动任何模型参数，仅通过优化外部 Harness 环境（文档结构优化、验证回路建立、约束机制设计），其编码智能体在 Terminal Bench 2.0 的得分从 52.8% 提升至 66.5%，排名从第30位跃升至第5位。这一案例成为"瓶颈在基础设施，不在模型智能"的最有力佐证。
+13.7% 任务完成率提升
第30→第5 排行榜跃升
Cc
Claude Code（源码泄露分析）
生产级 Harness 设计模式揭秘
Claude Code 意外暴露的 51.2 万行 TypeScript 源码揭示了生产级 Harness 的核心设计模式：三层记忆架构（轻量指针 MEMORY.md + 持久化存储 + 后台自动整合 autoDream）；插件式工具系统（40+ 个权限门控工具如 FileReadTool、BashTool）；子 Agent 模式（Fork 完全隔离、Teammate 协作、Worktree Git 隔离）；失败熔断机制（连续 3 次失败即停止重试）。
51.2万行 泄露源码规模
40+ 权限门控工具
V
Vercel
"约束优于能力"的典范
Vercel 的实践验证了一个反直觉的发现：削减 80% 的 Agent 工具反而实现了更少步骤、更低 Token 消耗和更快响应。这与 Can Bölük 的实验相呼应——仅调整工具格式（patch → hashline），同一模型的任务完成率从 6.7% 提升至 68.3%。
-80% 工具精简幅度
↓Token↓延迟 核心收益
M

十大核心工程实践

基于 OpenAI、Stripe、Anthropic、LangChain、Mitchell Hashimoto 等团队的一线经验，整理出 Harness Engineering 十大可落地实践：

1
AGENTS.md：文档作为免疫系统
每次 Agent 犯错，就在 AGENTS.md 中增加一条规则。使用渐进式披露结构，100行索引文件指向深层文档。每行规则都有具体的历史失败案例支撑。命令优先，代码示例优于文字说明。
2
架构约束机械化执行
通过 Linter 和 CI 门禁机械强制执行架构规则。定义严格的分层依赖方向并强制检查。Linter 错误消息必须包含修复指令，让 Agent 能自主修正。
3
闭环自验证（Build-Verify Loop）
规划→构建→验证→修复四步循环。强制 Agent 在退出前对照任务规范验证结果。采用"三明治推理"：规划（高推理）→实现（中推理）→验证（高推理）。
4
让应用对 Agent 可见
集成 Chrome DevTools Protocol 支持 DOM 快照和截图；为每个 git worktree 提供独立的日志、指标和追踪管道；让 Agent 能"看到"运行时状态，实现自主诊断。
5
长时运行 Agent 的会话间记忆
采用双体架构（初始化 Agent + 编码 Agent）。JSON 格式结构化记录进度，避免意外覆盖。每次会话先读取 Git 日志和进度文件，再开始新任务，实现状态连续性。
6
熵治理（对抗 AI Slop）
将架构规则直接写入仓库，由 Linter 强制执行。后台清理 Agent 定期扫描并自动提交重构 PR。持续小额偿还技术债务，避免集中爆发。
7
并行管理与人类注意力分配
稀缺资源从"计算"变为"人类注意力"。有人值守并行：同时运行少量 Agent，在自然休息时查看结果。无人值守并行：依赖成熟的 CI 和测试套件，实现全自动 PR 流程。
8
Blueprint 模式：确定性与概率性交替
在确定性代码节点（Linter、CI）和弹性 Agent 循环之间交替工作。确定性门禁夹住概率性工作，Agent 最多尝试2轮 CI，成功或放弃，避免无限重试。
9
Trace 驱动的迭代改进
基于实际运行轨迹（trace）优化 Harness，而非"凭感觉"调整。分析 LangSmith 等工具中的 traces，识别失败模式并针对性改进。建立吞吐量、质量、人类注意力消耗等度量体系。
10
渐进式采纳路径（Mitchell Hashimoto 六步）
① 停止用聊天机器人编码，转向能执行程序的 Agent；② 手动做一遍再用 Agent 重做；③ 下班前启动 Agent 做探索性工作；④ 将确定性任务外包给 Agent；⑤ 每次错误后更新 AGENTS.md；⑥ 始终有 Agent 在后台运行。

harness-creator 引擎

研究总结

核心结论
Harness Engineering 代表了软件工程在 AI 时代的根本性转变：从"如何让 AI 更智能"到"如何让 AI 可靠工作"。它不是一套具体的工具或框架，而是一种系统性的工程哲学——通过优化 AI 运行的环境，而非不断更换更强的模型，来实现可预测、可扩展、可信赖的 AI 系统。
核心等式 Agent = Model + Harness，模型提供智能，Harness 让智能落地为可用的系统
六层架构体系（L1 信息边界 ~ L6 约束恢复）是构建可靠 Agent 的完整设计蓝图
四大护栏（上下文工程、架构约束、反馈循环、熵管理）是最可落地的实践框架
LangChain 案例证明：纯 Harness 优化即可带来排名第30→第5的质的飞跃
工程师角色从"代码编写者"升级为"AI 系统的架构师与治理者"
知识工程是企业护城河，私域知识的沉淀决定 AI 应用的深度与壁垒
每当 Agent 犯错，就是改善环境设计的机会——Harness 在实践中不断进化
